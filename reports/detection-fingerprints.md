# Fingerprints of bad inference, visible from the outside

> Generated by `scripts/make_fingerprints.py` from [`findings/fingerprints.json`](../findings/fingerprints.json). Do not edit by hand.

Fingerprints of bad inference serving that a READER can look for in a published paper's own output — figures, tables, appendix transcripts, released JSON/JSONL, leaderboard entries, committed run logs. This is the evidence side of the survey: findings/survey.json says what each repo's CODE leaves open; this file says what could ever be SEEN from the outside. Inspired by nostalgebraist's catch of 'token soup' in the transcripts published with arXiv:2510.27338.

## The bottom line

Most published papers release nothing a reader could check. Of the fingerprint families below, exactly one (F2) can prove a routing problem outright, and it only works when the authors happened to save the raw response objects. Everything else is either suggestive-at-best or requires re-running inference. This is the expected result: quantization is engineered to be hard to notice, and aggregate-only reporting destroys every distributional signal before publication.

**17** fingerprint families survived. **4** have high discriminative power; **13** rest on a documented catch rather than on inference; **12** are checkable only when the authors released raw outputs.

## How this was built

18 independent web-research sweeps, each assigned a distinct detection angle, followed by one adversarial verifier per candidate whose job was to refute it (check every citation loads and says what was claimed; enumerate benign explanations; disqualify anything that secretly needs new inference). Merged into families by hand.

- raw candidates: **101** from independent search angles
- adversarially verified: **67**, of which **51** survived
- not verified: **34**

33 verifier agents and the synthesis agent were killed mid-run by an account spend limit. Families marked verification: 'partial' rest partly on sweep-agent claims whose citations were never independently re-checked; treat their real_instances as leads, not as settled citations. Families marked 'verified' rest only on candidates that passed adversarial citation checking.

## Reader's checklist

Cheapest and highest-yield first.

1. Does the paper name a provider, engine, precision, or dated snapshot for each model? If not, note that the number is a fact about an unspecified endpoint (F16).
2. Did they release raw outputs at all? If not, stop — nothing below is checkable, and say so plainly rather than implying the result is clean.
3. Grep the release for a provider/endpoint/generation-id key and count distinct values per reported condition. More than one is proof of mixed backends (F2).
4. Grep every completion field for literal special tokens and U+FFFD (F4).
5. Read a sample of the longest and the flagged-anomalous transcripts by eye, looking for topically unrelated word salad rather than merely hard-to-follow reasoning (F1).
6. Cross-tabulate empty-answer and finish_reason=length rates against provider and against the declared max_tokens (F3).
7. For reasoning models, check whether the reasoning-field shape is uniform across rows (F5).
8. For any parameter the paper says it set, check whether the released data shows it having had an effect, and check the endpoint's supported_parameters before concluding (F6).
9. For multi-sample releases, compute duplicate rate per model and compare within the release (F7).
10. Look up the model+benchmark on an independent per-provider comparison and see where the paper's number falls (F9).
11. Do NOT treat run-to-run variance, temperature-0 nondeterminism, repetition loops in isolation, coherent language-switching, or a cliff in a long-context curve as evidence (F12, F8, F1, F14).

## What you cannot tell, and why

This is a first-class result, not a caveat. Quantization is engineered to be subtle, and the reporting conventions of the field destroy most of the signal before publication.

- The typical paper publishes aggregate scores and nothing else. With no released generations, no per-item labels and no logprobs, every transcript- and distribution-based family above (F1, F3, F4, F5, F7, F8, F10, F12) is not merely unchecked but uncheckable.
- Single-run point estimates with no dispersion — the modal reporting style — make it impossible to distinguish a serving-induced shift from sampling noise, because there is no noise estimate to compare against.
- A 1–3 percentage-point accuracy shift, which is the size of effect moderate fp8/int8 quantization typically produces, is smaller than the swing caused by prompt template, few-shot ordering, or answer-extraction regex. It is invisible in a table without error bars and is not attributable even when visible.
- The strongest formal test (F10) needs roughly 10 completions per prompt from BOTH the suspect endpoint and a trusted reference on shared prompts. Almost no paper that merely uses an API model releases data of that shape; the precondition is met essentially only by papers auditing provider fidelity in the first place.
- Near-identical aggregate accuracy can hide large item-level churn — the same score with different items right. Detecting that needs per-item outputs from both conditions, which by definition means the comparison was already run.
- Which quantization was actually served is never recoverable from output text alone. Models disclose broad family identity at best and essentially never precision.
- Everything in F13 (model substitution) requires live access to the endpoint during the window in question. Once the window closes, a reader cannot re-run the probe.
- The motivating catch itself was not made from the paper's numbers. It required the authors having released raw transcripts AND a reader spending money re-running the experiment on a second provider. Absent either, the finding stands.

### F1 — Token soup — off-topic word salad in released transcripts

*Reasoning or answer text degenerates into a jumble of semantically unrelated words and non-dictionary fragments that has nothing to do with the prompt.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M1, M3, M8 |
| **Citation status** | verified |

**What you see.** Inside an appendix transcript or a released raw completion, spans of text that are not prose in any language: fused non-dictionary portmanteaus ('takeoffwhetherdenumg'), proper nouns with no connection to the topic dropped into technical content, orphaned diacritic fragments, sometimes several unrelated scripts jammed together. The tell is topical unrelatedness — the corrupted span uses vocabulary that has nothing to do with the question being answered.

**Where to look.** Appendix qualitative-example boxes reproducing raw model output verbatim; released completions JSONL; any figure that quotes chain-of-thought text.

**Why routing produces it.** A badly-configured or aggressively-quantized backend produces a next-token distribution flat enough that decoding wanders off the model's actual manifold. Because it is a decode-level failure rather than a reasoning failure, it appears in ordinary answer text too, not only in chain-of-thought.

**How to check.** Extract the exact quoted excerpt. Ask whether the incoherent span is (a) topically unrelated word salad, or (b) dense on-topic jargon / another language / compressed shorthand. Only (a) is a red flag. Then check whether the same corruption appears outside the chain-of-thought — corruption confined to CoT is more likely a property of the model's reasoning, corruption in ordinary answer text points at the decode path. If raw data is released with a provider field, check whether the flagged rows concentrate on one provider.

**Benign explanations — read these before concluding anything.**

- Dense, genuinely difficult content (SMILES notation, heavy math, compressed shorthand) reads as 'illegible' to a naive reader or an automated judge without being corrupted at all.
- Vendor-documented multilingual code-switching: DeepSeek-R1 and QwQ-32B-Preview both document language-mixing as a known, unresolved property. Coherent multi-language text is NOT this fingerprint; sub-word fragment salad that parses as no language at all is.
- A model reasoning under genuine difficulty produces clipped, jargon-dense text that scores as mildly illegible on an automated metric.

**Documented instances.**

- nostalgebraist read the chain-of-thought transcripts published with the illegible-CoT paper, recognized the corruption, re-ran the same prompts on a different OpenRouter provider (Novita rather than Targon), and found the illegibility disappeared. The paper's author conceded.  
  <https://www.lesswrong.com/posts/jHnZzicKzczkCCArK/r1-cot-illegibility-revisited>
  > a nonsensical melange of unrelated words/strings that looks almost like what you'd get if you picked each token uniformly at random from the model's vocab
- A second reader independently hit the same Targon-served corruption on their own re-run, before reading nostalgebraist's post.  
  <https://www.lesswrong.com/posts/WbP39ncim9hBsYn5t/what-counts-as-illegible-reasoning>

### F2 — Provider metadata surviving in the released data

*The raw response objects the authors saved still name which backend served each call — and more than one name under a single reported condition proves mixed backends outright.*

| | |
| --- | --- |
| **Discriminative power** | ●●● high |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M3, M4, M11 |
| **Citation status** | partial |

**What you see.** A released JSON/JSONL of raw responses carries a provider-identifying key — OpenRouter's top-level `provider` string, a harness field like `openrouter_provider`, or a `gen-…` generation id. Tabulating that key across all rows nominally belonging to one model in one experiment returns more than one distinct value. Weaker variants: the key is null on a fraction of rows; the key appears only in later runs of the same project; the committed config allow-lists several providers but every logged row names one.

**Where to look.** Released raw-response dumps, `inference.json`-style run artifacts, any per-request log committed to the repo. Not the PDF.

**Why routing produces it.** This is not a behavioral inference — it is the routing decision itself, recorded. OpenRouter's default routing load-balances across providers, and any harness that persists the response object rather than just the message text captures which one won each request.

**How to check.** Download the released file and group rows by the intended-fixed model, then count distinct provider values. More than one under a single reported condition means the condition was not served by one stack. Also count nulls: rows without the field cannot be audited either way, which is itself worth stating. Then diff the observed values against whatever the methods section or committed config claims. Caveat before you grep: OpenRouter's current published response schema does not document a top-level `provider` field (verified against the API reference, 2026-07-21), though real captures from 2025 carry one — so its absence in a release may reflect the API's shape at collection time rather than sloppiness by the authors.

**Benign explanations — read these before concluding anything.**

- Mixed providers do not by themselves prove the numbers are wrong — only that they did not come from one fixed serving stack, which is what most papers implicitly claim rather than something they assert and violate.
- A null provider field on a handful of rows can be ordinary retry noise rather than evidence of anything.
- Single-provider concentration is not inherently bad; it just means the number is a fact about that one endpoint, not about 'the model'.

**Documented instances.**

- nostalgebraist read the per-request provider field in the original study's released inference.json and established that, despite the config allow-listing two providers, every request had in fact gone to the weaker one.  
  <https://www.lesswrong.com/posts/jHnZzicKzczkCCArK/r1-cot-illegibility-revisited>
  > The OpenRouter providers listed in the config are targon/fp8 and Nebius, but in practice requests were always routed to Targon rather than Nebius, as can be confirmed by reviewing the inference
- A second reader auditing the same released data found rows with no provider metadata at all, which blocks the audit for those rows in either direction.  
  <https://www.lesswrong.com/posts/WbP39ncim9hBsYn5t/what-counts-as-illegible-reasoning>
  > The original data was generated with the 'Targon' provider (although some of the data does not have the provider metadata available).

### F3 — Truncation signatures — long reasoning, empty answer

*Rows where a long chain-of-thought trails off mid-clause next to an empty final-answer field, at a rate that differs by backend.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M2, M3, M8 |
| **Citation status** | verified |

**What you see.** In released raw completions for a reasoning model, a nontrivial fraction of rows pair a long, non-terminating reasoning field with a null/empty answer field. Alongside it: a completion-length histogram that piles up hard against a round number, or an elevated `finish_reason == "length"` rate concentrated in one condition.

**Where to look.** Released completions JSONL with separate reasoning and answer fields; any per-call log with a token-count or finish_reason column.

**Why routing produces it.** OpenRouter endpoints for one model slug advertise very different `max_completion_tokens`. An unpinned run therefore draws a mixture of output ceilings, and a reasoning model that overruns the smaller one is cut off before it writes an answer. The answer is then scored wrong, so the ceiling shows up as a capability difference.

**How to check.** Compute the empty-answer rate and cross-tabulate against the provider field and against completion length near the requested cap. Check whether the reasoning text on those rows ends mid-word rather than at a conclusion. Compare the histogram's wall against the max_tokens the methods section declares — if they match, that is innocent; a wall lower than what was requested is not.

**Benign explanations — read these before concluding anything.**

- Answer-extraction regex failures produce 'empty answer' rows even when the model did answer.
- RL-trained reasoning models genuinely loop and ramble on hard inputs without terminating, at any precision.
- A hard cliff at a round number is the default signature of an ordinary declared max_tokens setting in essentially every eval harness — check the methods section before suspecting anything.

**Documented instances.**

- Re-running the illegible-CoT experiment unpinned split traffic across two providers; one of them frequently ran out of output budget mid-reasoning.  
  <https://www.lesswrong.com/posts/jHnZzicKzczkCCArK/r1-cot-illegibility-revisited>
  > After reviewing the results, I found that responses from Azure frequently had empty final answer fields, suggesting that they ran out of room while still writing CoT.

### F4 — Special-token and chat-template litter

*Turn-boundary tokens, template debris, or replacement characters printed as ordinary text in the released output.*

| | |
| --- | --- |
| **Discriminative power** | ●●● high |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M3, M8, M11 |
| **Citation status** | verified |

**What you see.** Literal `<|eot_id|>`, `<|start_header_id|>assistant`, `<|im_start|>`, `<|end_of_text|>`, `</s>`, or `<|endoftext|>` sitting in running prose; a stray hallucinated `assistant`/`System:`/`User:` turn appended after a complete answer; a system prompt rendered as a Python list-literal repr; U+FFFD (�) inside otherwise normal text.

**Where to look.** Any released completion field, or appendix boxes that reproduce raw output verbatim rather than re-typesetting it.

**Why routing produces it.** The serving stack's stop/EOS configuration or chat template does not match the model's. The Llama-3 family lists several valid EOS ids and several backends historically registered the wrong one, so the model emits its trained turn boundary and the stack prints it instead of consuming it. Quantization and routing are upstream causes here (a different provider is a different engine with a different generation_config), not the observable itself.

**How to check.** Grep every released completion field for the literal special-token substrings, case-sensitive and unescaped. Check they are not inside a fenced example the authors are deliberately discussing. If the release has a provider field, check whether the hits cluster on one endpoint — that is close to proof. For U+FFFD, check `finish_reason` first: truncation mid-multibyte-character is the mundane cause and is very common.

**Benign explanations — read these before concluding anything.**

- Papers frequently typeset special tokens on purpose in appendix boxes to illustrate their prompt format or a jailbreak payload.
- A prompt-injection or template-confusion payload can make the model echo literal special-token strings back.
- U+FFFD from truncation mid-character happens on any correctly-served model whenever generation is cut off.
- Base (non-instruct) or small models hallucinate chat-markup-looking text on their own.

**Documented instances.**

- A user serving Llama-3-8B-Instruct and Mistral-7B-Instruct saw turn-boundary tokens appear as literal text; a maintainer traced it to misconfigured role/stop handling.  
  <https://github.com/guidance-ai/guidance/discussions/786>
  > I keep getting either `</s>` (for Mistral) or `<|eot_id|><|start_header_id|>assistant` for llama-3-8b-instruct in my outputs.
- A Llama-3 GGUF conversion embedded the wrong EOS id, producing output that never stopped.  
  <https://huggingface.co/gradientai/Llama-3-8B-Instruct-262k/discussions/13>
  > Llama-3 uses 128009 as the correct EOS token ID. However, 128001 is entered in the GGUF file. So this can't work.
- The same class of mismatch on Llama-3-70B-Instruct, described from the reader's side of the output.  
  <https://huggingface.co/turboderp/Llama-3-70B-Instruct-exl2/discussions/1>
  > it kept adding word 'assistant' at the end of each message, or failing to stop and continuing until running out of the token limit.

### F5 — Reasoning-field shape inconsistency

*For one nominal reasoning model in one release, the chain-of-thought sometimes sits inline in `<think>` tags, sometimes in a separate `reasoning` key, and sometimes is absent entirely.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M2, M3, M8 |
| **Citation status** | verified |

**What you see.** Non-uniform shape of the reasoning-bearing field across rows of the same model and condition. Related: per-example reasoning-token counts that vary wildly under a fixed requested reasoning effort, in a way that tracks time or backend rather than problem difficulty.

**Where to look.** Released completions JSONL for reasoning models; any table reporting reasoning-token counts.

**Why routing produces it.** Different backends surface reasoning differently through the same router — inline-tagged, separately-keyed, or suppressed unless an opt-in flag is set. `reasoning_effort` and thinking budgets are among the parameters providers may accept and ignore. A paper that measures a property of the chain-of-thought is maximally exposed to this, because the object being measured is the one the serving stack reshapes.

**How to check.** For a fixed model in a released dataset, check whether the reasoning field's shape is uniform across all rows. Non-uniformity correlating with a provider or endpoint id is the tell. For reasoning-length claims, check whether the variance tracks difficulty (benign) or backend/time (not).

**Benign explanations — read these before concluding anything.**

- A model can legitimately produce little or no chain-of-thought on an easy question.
- Reasoning-token variance is usually intrinsic to problem difficulty.
- Claude's 'reasoning' exposed through any API is a summary by design, documented by the vendor, and applies equally to direct callers — a difference in texture from R1's raw trace is product behavior, not corruption.

**Documented instances.**

- DeepSeek-R1 served via OpenRouter returns reasoning inline inside `<think>` tags in the content field, whereas the native API separates it.  
  <https://github.com/OpenHands/OpenHands/issues/6600>
  > the reasoning content is prepended to the actual response message, inside the `<think>` tag

### F6 — Requested parameters that visibly did nothing

*A decoding or format parameter the paper says it set has no effect on some rows — the strongest behavioral evidence available, because a model does not choose to ignore its own sampler.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M2, M7, M9 |
| **Citation status** | verified |

**What you see.** A top_p / top_k / frequency_penalty / min_p sweep whose generations are byte-identical across settings for a subset of rows while other rows respond normally. Or: JSON-schema validation failing on an uneven subset of rows where `response_format` was set. Or: logprobs populated on some rows and null on others for one model. Or: completion lengths far exceeding the declared max_tokens, or stop sequences plainly not honored.

**Where to look.** Ablation tables and figures; released structured-output datasets; any released file with a logprob or token-count column.

**Why routing produces it.** OpenRouter forwards requests to providers that do not support every parameter and, unless `require_parameters: true` is set, those providers silently ignore what they cannot honor. The decoding configuration the paper reports is then not the configuration that ran.

**How to check.** Before concluding anything, pull the model's entry from OpenRouter's models/endpoints API and read `supported_parameters` for the parameter in question — a documented non-support is an ordinary limitation, not a routing accident. Then check whether the null-effect rows cluster by provider or time window. Note that at temperature ≈ 0 a null top_p effect is expected and proves nothing.

**Benign explanations — read these before concluding anything.**

- Some models and providers simply never support the parameter, statically and documented — the harness may correctly fall back.
- Baseline JSON-validity differs enormously by model regardless of provider quality; OpenRouter's own telemetry shows per-model failure rates from ~0.4% to ~12%.
- Stop-sequence overruns are more often caused by a stop string that never appears in the model's true tokenization than by provider-side dropping.
- At near-deterministic temperature, several sampling parameters correctly have no visible effect.

**Documented instances.**

- A systematic probe of every reachable OpenRouter endpoint found logprob support varies across endpoints for the same nominal model, and that the router itself imposes restrictions absent from the vendor's direct API.  
  <https://arxiv.org/abs/2512.03816>
  > In the case of OpenAI, we found that requesting less than 16 tokens of output from the GPT-4.1 series is not supported on OpenRouter, but only on the OpenAI API
- OpenRouter published pre-fix JSON-validity rates for structured-output requests broken down by model, establishing the model-level baseline a reader must compare against.  
  <https://openrouter.ai/blog/announcements/response-healing-reduce-json-defects-by-80percent>
  > Gemini 2.0 Flash (1.62M requests): Success Before: 99.61%, Success After: 99.92%
- A parameter-translation audit of a deployed multi-provider gateway found top_p, frequency/presence penalties and top_k were never forwarded at all on some paths.  
  <https://github.com/Lore-Hex/quill-router/issues/130>
  > top_p is dropped even though it is trivially supported by every OpenAI-compatible provider, so nucleus-sampling control silently no-ops there.

### F7 — Entropy collapse across nominally independent samples

*Repeated draws at temperature > 0 come back identical far more often than the requested sampling temperature could explain.*

| | |
| --- | --- |
| **Discriminative power** | ●●● high |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M2, M7 |
| **Citation status** | verified |

**What you see.** In released per-prompt multi-sample data (pass@k, self-consistency), an anomalous fraction of 'independent' draws for one model or provider are identical or near-identical strings, while other models in the same release show normal variation on the same prompts.

**Where to look.** Released multi-sample JSON/JSONL with several completions per prompt id; self-consistency or majority-vote appendices.

**Why routing produces it.** Response caching (a memoized answer served regardless of seed) or a stack that ignores the requested temperature and serves near-greedy output while the request is labeled as sampled.

**How to check.** Compute a duplicate-rate or distinct-n statistic over the released per-prompt sample sets, and compare it across providers and models WITHIN the same release on identical prompts. The signal is a within-release outlier, not an absolute threshold you invent. Do not run this on a paper's handful of illustrative appendix samples — the N is far too small.

**Benign explanations — read these before concluding anything.**

- Short factual, numeric or heavily-constrained prompts genuinely have low answer entropy; duplicates are expected.
- Without a reference distribution for the same prompt type, an absolute duplicate rate is uninterpretable.
- Appendix excerpt boxes have too few samples to distinguish collapse from coincidence.

**Documented instances.**

- Model Equality Testing compared repeated samples from a commercial Llama endpoint against reference weights at requested temperature 1 and found the endpoint's repeated draws far less varied than the reference.  
  <https://arxiv.org/pdf/2410.20247>
  > some APIs (in particular, Perplexity) seem to be producing lower-entropy completions than the reference distribution, suggesting some form of caching or incorrect implementation of the temperature parameter.

### F8 — Degenerate repetition and single-glyph collapse

*Completions that collapse into a repeated character, word, or phrase instead of an answer — real under aggressive quantization, but also the classic failure of perfectly-served models.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M1, M12 |
| **Citation status** | verified |

**What you see.** A completion that is one short word followed by dozens of repeated punctuation characters, a stuck n-gram cycle, or in the extreme case a single non-English glyph repeated to the output limit for an ordinary English prompt.

**Where to look.** Appendix qualitative-sample boxes; released completions on code or knowledge tasks; the tail of long generations.

**Why routing produces it.** Weight-only quantization to nf4/int4 degrades the next-token distribution most on harder prompts and can collapse decoding into a loop. Uncalibrated FP8 recipes have produced total collapse. But EOS/stop-handling bugs produce visually identical output with no quantization involved.

**How to check.** This is only usable as a PAIRED comparison: the same prompt, the same nominal model, two serving conditions shown side by side, one degenerate and one not. Repetition in isolation is not evidence. Also check whether the paper set a repetition penalty and whether its stated provider honors it.

**Benign explanations — read these before concluding anything.**

- Neural text degeneration under greedy or low-temperature decoding is well documented at full precision, especially on long or out-of-distribution generations.
- Open-source reasoning models mode-collapse at low temperature by their own nature: one study documents up to ~4% degenerate-repetition rates on AIME/GPQA even with a repetition penalty correctly applied to an unquantized model.
- A broken prompt template, missing stop token, or sampler misconfiguration produces identical-looking output.

**Documented instances.**

- Model Equality Testing compared an nf4-quantized local copy of Llama-3.1 against a commercial endpoint on the same code prompts; the quantized copy produced degenerate repeated-character completions.  
  <https://arxiv.org/pdf/2410.20247>
  > we observe that the Llama 70B-scale models quantize poorly, and their nf4 completions are degenerate ... Perplexity answers with incorrect code snippets, while nf4 outputs nonsense completions (e.g., "simulation!!!!!!!!!![...]").
- An uncalibrated FP8 quantization recipe made a small chat model answer every English prompt with one repeated Chinese glyph.  
  <https://github.com/mlc-ai/mlc-llm/issues/2579>
  > When FP8 quantization is applied to the TinyLlama-1.1B-Chat-v1.0 model, the model responses are exclusively composed of the simplified chinese character `给`.
- A decoding-methods paper documents reasoning models collapsing into single-token cycles at low temperature on a correctly-served, unquantized instance — the benign baseline.  
  <https://arxiv.org/abs/2504.20131>
  > Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! Non! nono! This! This! This! This!

### F9 — The number sits inside a known cross-provider spread

*An open-weight model's reported score is compared against an independent per-provider breakdown of the same model on the same benchmark — and the spread is wider than the effect the paper claims.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | no |
| **Reveals** | M1, M3, M8, M10 |
| **Citation status** | verified |

**What you see.** A paper reports one number for an open-weight model with no provider named, while a public per-provider comparison for that model and benchmark spans tens of points. The paper's number may sit near the bottom of that range, or outside it entirely.

**Where to look.** The paper's results table, cross-referenced against Artificial Analysis provider comparisons, Aider's quantization post and leaderboard, or a vendor's own provider-verification table.

**Why routing produces it.** Providers differ in quantization, engine, context window, default sampling, and tool-call implementation. One documented case bundled a stale engine build that ignored `reasoning_effort`, a deliberately compressed model variant, and ordinary engine differences under one model name.

**How to check.** Look up the model+benchmark on an independent per-provider comparison. If the paper names and pins a provider, see where that provider falls. If it names none, the number is unverifiable and could be anywhere in the documented spread — say that. Note these dashboards are live and providers patch: a spread measured last year may have closed.

**Benign explanations — read these before concluding anything.**

- Small benchmarks give wide confidence intervals; part of any spread is sampling noise.
- Harness differences (prompting, parsing, requested effort) between the paper and the reference dashboard explain gaps with no serving story needed.
- Providers may legitimately differ in prompt-template handling.
- Reasonable quantization (fp8, high-bit AWQ) often costs close to nothing; the published examples are the worst cases someone went looking for.

**Documented instances.**

- Artificial Analysis ran AIME25 across ~19 providers serving gpt-oss-120b; scores ranged from 93.3% to 36.7%. Microsoft's engineer confirmed the cause of one low score.  
  <https://simonwillison.net/2025/Aug/15/inconsistent-performance/>
  > This is exactly it, it's been fixed as of yesterday afternoon across all serving instances (of the hosted 120b service). Old vLLM commits that didn't respect reasoning_effort, so all requests defaulted to medium.
- Aider benchmarked one coding model across hosting providers and found a ~20-point spread, plus a separate near-zero outlier, traced partly to a provider silently truncating context.  
  <https://aider.chat/2024/11/21/quantization.html>
  > Ollama does not throw an error if you submit a request that exceeds the context window. Instead, it just silently truncates the request by discarding the 'oldest' messages in the chat to make it fit within the context window.

### F10 — Two-sample distributional tests on released completions

*The strongest formal method for asking 'is this the model they claim' — and it almost never applies, because it needs paired raw completions papers do not publish.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M1, M4, M8 |
| **Citation status** | verified |

**What you see.** Not a visual pattern. A property of the release: does it contain multiple raw string completions per prompt from two or more sources on a shared prompt set?

**Where to look.** Supplementary data or the code repository, not the PDF.

**Why routing produces it.** Model Equality Testing frames auditing as a two-sample test — MMD with a Hamming-distance string kernel over completions, significance by permutation. It needs roughly 10 completions per prompt over 20–25 prompts and no logprobs, and it flagged 11 of 31 commercial endpoints. A follow-up rank-based uniformity test is more robust to formatting artifacts.

**How to check.** Confirm the release actually contains paired multi-sample raw completions. If so, run the authors' released code directly on the files. If not — the overwhelmingly common case — this method needs wholly new data collection against a trusted reference, which is no longer reading a paper.

**Benign explanations — read these before concluding anything.**

- Formatting artifacts alone can trigger a false positive: one flagged endpoint differed only by omitting leading whitespace.
- Ordinary backend nondeterminism (batching, non-batch-invariant kernels) inflates the statistic.
- Any mismatch in temperature, top_p or max_tokens between the two arms will correctly detect 'different distributions' with no misconduct involved.

**Documented instances.**

- Gao, Liang and Guestrin tested 31 commercial endpoints across 9 providers serving 4 Llama models with an MMD-based test against reference weights.  
  <https://arxiv.org/html/2410.20247>
  > 11 out of 31 endpoints serve different distributions than reference weights released by Meta.
- A follow-up documented a false-positive mechanism in the naive Hamming-kernel approach.  
  <https://arxiv.org/html/2506.06975v1>
  > the Hugging Face Inference API consistently omits the leading whitespace present in the reference outputs

### F11 — Judge and parser failures that fall unevenly

*Parse-failure or dropped-sample rates that differ by arm, with the pipeline defaulting unparseable output to a floor score.*

| | |
| --- | --- |
| **Discriminative power** | ●○○ low |
| **Evidence** | `literature_supported` — the underlying degradation is published; the catch is not |
| **Needs raw outputs** | no |
| **Reveals** | M2, M9 |
| **Citation status** | verified |

**What you see.** A reported parse-failure rate that rises specifically for the arm that scores worse; an explicit statement that unparseable responses are scored 0; an n below the benchmark's canonical size with an infrastructure caveat; a cluster of empty or error-string completions in a released log.

**Where to look.** The methods paragraph where n is stated; footnotes about dropped or invalid samples; per-sample logs' error fields; the paper's own self-audit tables where it has them.

**Why routing produces it.** A judge or extractor that relies on structured output gets free-form text from a route that does not enforce the schema, or gets truncated text; the fallback scores it at the floor. Provider timeouts and rate limits drop samples from the denominator. Both push measured differences in a direction that correlates with which arm was served by the flakier route.

**How to check.** Check whether a per-arm parse-failure or truncation rate is reported at all. Compare the stated n against the benchmark's canonical size and see whether the gap is explained. If raw generations are released, re-run extraction with a permissive parser and see how much of the gap survives.

**Benign explanations — read these before concluding anything.**

- Exclusions are often legitimate and disclosed (malformed benchmark items).
- Agentic benchmarks have well-known execution-environment flakiness unrelated to which provider served the model.
- Smaller models really are worse at strict output formats; asymmetric parse failure can be a genuine capability difference.
- In the best-documented case the generation budgets were the authors' own disclosed choice, not a hidden provider default.

**Documented instances.**

- A methodology paper measured how format mismatch and truncation inflate apparent failure asymmetrically across model sizes.  
  <https://arxiv.org/abs/2605.07395>
  > Format mismatch creates asymmetric artifacts: smaller models (E2B: 12.4% parse failure on MMLU) are more affected than larger models (31B: 5.4%)
- Epoch AI's reproduction work found rate-limit failures concentrated on OpenRouter-routed endpoints for one open-weight model.  
  <https://epoch.ai/gradient-updates/why-benchmarking-is-hard>
  > Some providers return RateLimitErrors, especially when accessing GLM-4.6 over OpenRouter.

### F12 — NEGATIVE CONTROL — variance and temperature-0 nondeterminism are not fingerprints

*Wide run-to-run variance, and temperature-0 runs that do not reproduce, are normal on a single honest backend and must not be cited as evidence of routing problems.*

| | |
| --- | --- |
| **Discriminative power** | ●○○ low |
| **Evidence** | `literature_supported` — the underlying degradation is published; the catch is not |
| **Needs raw outputs** | yes |
| **Reveals** | M7 |
| **Citation status** | verified |

**What you see.** Repeated runs of one model under fixed settings swinging far more than binomial noise; temperature-0 or fixed-seed runs that fail to reproduce.

**Where to look.** Any paper reporting repeated runs; any reproducibility claim resting on temperature=0 or a seed.

**Why routing produces it.** Continuous batching, chunked prefill, prefix caching and MoE routing all make outputs non-reproducible on a single stack with no quantization or provider swap involved. Sampling 1000 completions at temperature 0 from one fixed serving stack does not return 1000 identical strings.

**How to check.** Do not use this as evidence. The only version with any power is a genuinely BIMODAL score histogram — two separated clusters, not one wide distribution — and even then, establish the model's own single-backend baseline variance first.

**Benign explanations — read these before concluding anything.**

- Engine-level nondeterminism produces this on a single honest backend.
- One study found a single-vendor proprietary API showing the same near-zero repeat agreement as open-weight models, so the observable does not even separate single-vendor from multi-provider serving.
- Higher-temperature and agentic tasks are intrinsically high-variance.

**Documented instances.**

- Thinking Machines Lab demonstrated temperature-0 nondeterminism on a single fixed serving stack with no router involved.  
  <https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/>
  > We use Qwen/Qwen3-235B-A22B-Instruct-2507 and sample 1000 completions at temperature 0

### F13 — Model-substitution tells

*Behavioral evidence that the endpoint was not serving the announced model at all — the rarest and most decisive case, and it needs live API access, not a paper.*

| | |
| --- | --- |
| **Discriminative power** | ●●● high |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | yes |
| **Reveals** | M4, M6 |
| **Citation status** | verified |

**What you see.** Generation halting on another vendor's control tokens; a vendor's name silently filtered out of responses; output-length ceilings matching a different tokenizer; self-identification contradicting the stated model.

**Where to look.** Raw completions obtained by actively probing a live endpoint. Not published papers — the flagship case had no paper at all.

**Why routing produces it.** A hosted endpoint forwards to a different model entirely. Ordinary quantization does not produce these tells; substitution does.

**How to check.** Requires live access during the relevant window. A reader of a published paper effectively cannot run this. Treat self-identification especially cautiously: it is trivially overridden by a system prompt, and open-weight models routinely misidentify themselves because of training-data contamination.

**Benign explanations — read these before concluding anything.**

- Self-misidentification is an extremely common artifact of distillation on another model's outputs, independent of serving.
- Reproduction failure alone is indistinguishable from ordinary quantization degradation and is never proof by itself.
- One audit found models disclose only broad family identifiers and almost never precision or quantization status.

**Documented instances.**

- Testers established that a hosted 'Reflection 70B' API was forwarding to a different vendor's model, using tokenizer-level behavioral tells rather than benchmark scores.  
  <https://x.com/RealJosephus/status/1832904398831280448>
  > "Reflection API" is a sonnet 3.5 wrapper with prompt. And they are currently disguising it by filtering out the string 'claude'.

### F14 — Context-window and prompt-compression artifacts

*A long-context result shaped by the endpoint's window rather than the model's — including OpenRouter silently deleting the middle of oversized prompts.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | no |
| **Reveals** | M3, M8, M10 |
| **Citation status** | verified |

**What you see.** A sharp, step-function accuracy cliff tied to a round total-length threshold for a model whose endpoint context length sits at or below it; or a response showing no trace of verbatim material that was supposedly in the prompt.

**Where to look.** Long-context accuracy-vs-length plots; needle-in-a-haystack heatmaps; any benchmark run through a router without transforms explicitly disabled.

**Why routing produces it.** OpenRouter documents automatically enabling a compression transform for endpoints at or below 8,192 tokens of context, and whenever a request would exceed the endpoint's window — removing content from the middle of the prompt rather than erroring.

**How to check.** Check whether the paper pinned a provider and disabled transforms. Cross-reference the endpoint's context length against the input lengths actually tested. A cliff by itself proves nothing; the cliff has to sit at a threshold with no architectural explanation for that model.

**Benign explanations — read these before concluding anything.**

- 'Lost in the middle' is a real, extensively documented property of correctly-served models.
- Near-vertical accuracy cliffs at arbitrary token counts are ordinary model behavior on direct-query studies with no router involved.
- RoPE-extrapolation and sliding-window boundaries produce sharp cliffs at full precision.
- Many models have a genuine trained 8k context that coincides with the compression trigger.

**Documented instances.**

- OpenRouter documents that a compression transform is enabled by default for small-context endpoints and silently removes content from the middle of oversized prompts.  
  <https://openrouter.ai/docs/guides/features/message-transforms>

### F15 — Capability asymmetries quantization causes and aggregate metrics hide

*Quantization does not degrade everything evenly — safety behavior, low-resource languages and long chains move while the headline metric barely does.*

| | |
| --- | --- |
| **Discriminative power** | ●●○ medium |
| **Evidence** | `literature_supported` — the underlying degradation is published; the catch is not |
| **Needs raw outputs** | no |
| **Reveals** | M1, M8, M10 |
| **Citation status** | verified |

**What you see.** A quantized checkpoint reported within ~2 points of full precision on quality metrics while its refusal rate swings enormously; a multilingual table with a 1–3% automatic-metric drop where the actual non-English generations are off-target or truncated; near-matching accuracy paired with substantially longer chains of thought.

**Where to look.** Quantization-ablation tables; per-language breakdowns; token-count columns reported next to accuracy; the actual released generations for the affected slice.

**Why routing produces it.** Alignment behavior is carried by a comparatively small set of directions that general quality metrics are statistically insensitive to. Low-frequency tokens have the least margin before quantization error flips a decision. Quantization noise degrades step-level confidence, so models re-derive steps rather than answer wrong.

**How to check.** Do not infer safety stability from a stable quality score — probe refusal behavior on the specific endpoint. For multilingual work, read a sample of the actual non-English generations rather than trusting the automatic metric. For reasoning-length claims, insist on a matched full-precision arm.

**Benign explanations — read these before concluding anything.**

- The quality/safety decoupling is not universal — in one study only about 20% of quantized checkpoints showed it, and the sign was inconsistent across families.
- Base models are already weaker on low-resource languages before any quantization.
- Tokenizer over-fragmentation of non-Latin scripts produces repetition and truncation independent of precision.
- Token inflation is concentrated at aggressive INT3/INT4 activation settings, not the fp8/int8 levels a commercial provider is most likely to serve.

### F16 — Paperwork tells — what the methods section does not say

*No provider, no quantization, no dated snapshot, and a long collection window: not evidence of corruption, but the precondition for it being undetectable.*

| | |
| --- | --- |
| **Discriminative power** | ●○○ low |
| **Evidence** | `documented_catch` — someone really caught a bad provider this way |
| **Needs raw outputs** | no |
| **Reveals** | M6, M10, M12 |
| **Citation status** | partial |

**What you see.** A one-sentence serving disclosure ('we evaluate models through OpenRouter') with no provider, engine, precision or dated snapshot; a bare undated model slug next to a stated multi-week collection window; `:free` or `:floor` slugs or `sort: "price"` in committed source; a config whose earliest committed version had no routing key at all.

**Where to look.** The methods paragraph, the README, and every committed run config — not just the one the README shows as an example.

**Why routing produces it.** None of this corrupts anything on its own. It is the reporting gap that makes corruption unfalsifiable, and it is close to universal: a 72-paper audit found missing model identity present in 73.6% of papers and partly present in a further 12.5%, with none of them discussing it.

**How to check.** Search the full paper text for any provider/snapshot/quantization/engine language. Diff the earliest committed config for the model against the one the headline run used. Grep configs for provider-tag suffixes (`/fp8`, `/int4`, `/fp4`) and for `:free`/`:floor`. Because this is near-universal, its presence in any single paper is weak on its own — it earns weight only in combination with an open-weight model on a critical result path.

**Benign explanations — read these before concluding anything.**

- Terse methods sections of exactly this shape are the norm, not the exception.
- Short collection windows make version drift a non-issue.
- `:free` is documented in terms of rate caps, not precision, so it does not by itself establish quantization.
- Single-run point estimates with no dispersion are standard practice driven by eval cost.

**Documented instances.**

- A systematic review of 72 peer-reviewed LLM-security papers at top venues found model ambiguity — no snapshot, commit, or quantization — to be the single most prevalent pitfall, and unacknowledged everywhere it appeared.  
  <https://arxiv.org/abs/2512.09549>
  > 73.6% (53) of all papers, andPartly presentin 12.5% (9)
- The author of the illegible-CoT paper conceded after a cross-provider re-run that the paper's serving had not been controlled.  
  <https://www.lesswrong.com/posts/jHnZzicKzczkCCArK/r1-cot-illegibility-revisited>
  > I should have been running capability evaluations with different providers to see which one was serving it properly

### F17 — NEGATIVE RESULT — cost and throughput side channels mostly do not work

*Per-token cost and tokens/sec look like they should identify the endpoint and its precision. Measured, they largely do not.*

| | |
| --- | --- |
| **Discriminative power** | ●○○ low |
| **Evidence** | `literature_supported` — the underlying degradation is published; the catch is not |
| **Needs raw outputs** | yes |
| **Reveals** | M4 |
| **Citation status** | partial |

**What you see.** A logged cost or latency column that seems to promise backend identification.

**Where to look.** Released per-request logs with cost, duration or throughput columns.

**Why routing produces it.** OpenRouter prices differ per provider, so cost per token is in principle close to an endpoint identifier. In practice measured cost-per-token was effectively identical across three different providers serving one model, and price ranks poorly against disclosed precision. Extreme throughput more often identifies specialised hardware than low precision — one provider serves a model roughly 40× faster than another at comparable quality.

**How to check.** Use cost only as weak corroboration after checking the provider field directly. Never read high throughput as a quantization red flag without identifying the provider first. Wall-clock mixes decode speed with queueing, retries, network jitter and (for reasoning models) how long the chain happened to run.

**Benign explanations — read these before concluding anything.**

- Cost per token was measured as near-identical across distinct providers of one model.
- Specialised-silicon vendors produce order-of-magnitude throughput differences at full precision.
- Latency is dominated by confounds that have nothing to do with serving quality.
